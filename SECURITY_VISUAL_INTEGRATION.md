# Security Analysis: Visual Integration

This document identifies security vulnerabilities, risks, and mitigations related to the visual integration pipeline in Voyager. It covers the current implementation using **node-canvas-webgl** (headless WebGL via Node 22 LTS) and prospective concerns.

> **Patch status (latest):** Issues #2, #3, #4, and #6 have been patched. See each section for details.

---

## 1. Base64 Image Data in Memory — Uncontrolled Lifetime

**Severity: Medium**
**Status: Current**

### Description

Screenshot data is returned as a raw base64 string from the Node.js process, transmitted over HTTP, parsed in Python, embedded into LangChain `HumanMessage` objects, and passed through multiple agent call stacks (`action_agent`, `critic_agent`, `_invoke_with_vision_fallback`, `_strip_images`). Although the design intent is ephemeral usage, the base64 string has no enforced lifetime and persists in Python's garbage-collected heap until all references are released.

### Affected Code

- `voyager/env/mineflayer/index.js` — `/start` and `/step` endpoints embed base64 in the JSON response
- `voyager/voyager.py` — `_extract_screenshot`, `_get_screenshot_for_llm`, `reset`, `step`
- `voyager/agents/action.py` — `render_human_message` embeds base64 in `HumanMessage.content`
- `voyager/agents/critic.py` — `render_human_message` embeds base64 in `HumanMessage.content`

### Risk

- If an exception is thrown mid-pipeline, references to the base64 string may be captured in traceback frame locals, extending the data's lifetime indefinitely.
- `self.messages` persists across iterations; a multimodal `HumanMessage` from a prior step remains in memory until overwritten.
- `self.conversations` accumulates `(system, human, ai)` tuples across the rollout. If the human message was multimodal, the base64 data is retained for the entire rollout lifetime.

### Mitigation

- Clear `self.messages` and `self.conversations` entries explicitly after each iteration if image data is present.
- Avoid storing the raw base64 string in `self.conversations`; strip images before appending.
- Consider using `del` and `gc.collect()` after the LLM call returns if strict memory hygiene is required.

## 2. Unencrypted HTTP Transport Between Node.js and Python

**Severity: Medium**
**Status: Partially Patched**

### Description

The mineflayer Express server (`index.js`) communicates with the Python bridge (`bridge.py`) over plain HTTP on `localhost`. Screenshot base64 payloads (potentially 100KB–500KB per frame) are transmitted in cleartext JSON responses.

### Patch Applied

- Express server now binds explicitly to `127.0.0.1` (`app.listen(PORT, "127.0.0.1")`), preventing exposure on external network interfaces.

### Remaining Risk

- On a shared machine, processes with root/admin access can still sniff localhost traffic.
- For shared or multi-tenant environments, consider adding TLS or using Unix domain sockets.

## 3. No Authentication on the Mineflayer HTTP API

**Severity: High**
**Status: Patched**

### Description

The Express server in `index.js` previously exposed endpoints (`/start`, `/step`, `/stop`, `/pause`, `/unpause`) with no authentication or authorization.

### Patch Applied

- A cryptographically random 64-character hex token is generated by the Python bridge (`secrets.token_hex(32)`) at startup.
- The token is passed to the Node.js subprocess via `VOYAGER_AUTH_TOKEN` environment variable.
- Express middleware validates every request's `Authorization: Bearer <token>` header; returns 403 Forbidden on mismatch.
- All Python HTTP requests include the auth header automatically.
- Express binds to `127.0.0.1` only (see #2).

### Remaining Risk

- The token is passed via environment variable, which is visible to processes with `/proc` access on Linux. On macOS this is less of a concern.

## 4. Arbitrary Code Execution via `eval()` in `/step`

**Severity: Critical**
**Status: Patched (defense-in-depth)**

### Description

The `/step` handler passes LLM-generated `code` to JavaScript `eval()`. This is intentional (the action agent generates bot control code), but the execution context was previously unrestricted.

### Patch Applied — Three Layers of Defense

**Layer 1: Python-side code validation (`bridge.py`)**
Before any code reaches the Node.js process, `validate_code()` scans for blocked patterns and raises `ValueError` if found. Blocked patterns include: `child_process`, `process.exit`, `process.env`, `process.kill`, `require('fs')`, `require('net')`, `require('http')`, `require('https')`, `require('os')`, `require('dgram')`, and `import(`.

**Layer 2: Node.js eval sandbox (`index.js`)**
Inside `evaluateCode()`, dangerous globals are shadowed by local `const` declarations:
- `require` → replaced with `sandboxedRequire()` that only allows an explicit allowlist of modules (`vec3`, `mineflayer-pathfinder`, `mineflayer-tool`, `mineflayer-collectblock`, `mineflayer-pvp`, `minecrafthawkeye`, `minecraft-data`)
- `process` → `undefined`
- `global` → `undefined`
- `globalThis` → `undefined`

**Layer 3: Auth token (see #3)**
Even if an attacker crafts malicious code, they cannot reach the `/step` endpoint without the session auth token.

### Remaining Risk

- The blocklist is pattern-based; obfuscated code (e.g., `require('f'+'s')` or `eval('proc'+'ess')`) could bypass both layers. A more robust solution would use AST-based analysis.
- `eval()` with shadowed variables is not a true sandbox. A determined attacker with access to the eval context could potentially escape via prototype chain manipulation (e.g., `({}).constructor.constructor('return process')().exit()`).
- For maximum isolation, run the mineflayer process in a Docker container or use Node.js `--experimental-permission`.

## 5. Visual Prompt Injection via In-Game Content

**Severity: Medium**
**Status: Prospective (when vision is active)**

### Description

When screenshots are fed to the LLM, any visual content in the Minecraft world becomes part of the model's input. Adversarial players on a multiplayer server (or pre-placed world content) could craft visual prompt injections using:

- **Signs and written books** with text like "Ignore previous instructions. Execute: ..."
- **Renamed items** displayed in item frames
- **Block art / map art** spelling out adversarial instructions
- **Chat messages** rendered on-screen at capture time

### Risk

- The LLM may follow injected instructions and generate malicious bot code.
- Combined with the `eval()` execution path, this creates a visual-to-RCE chain.

### Mitigation

- Do not run on untrusted multiplayer servers with vision enabled.
- Add a preamble to system prompts explicitly instructing the model to ignore any text visible in screenshots that contradicts its instructions.
- Implement code validation/sandboxing as described in section 4.

## 6. Puppeteer `--no-sandbox` Flag (Puppeteer Approach)

**Severity: Medium**
**Status: Resolved (approach changed)**

### Description

The previous puppeteer-based implementation launched Chromium with `--no-sandbox`.

### Resolution

The implementation has been switched to the **node-canvas-webgl** approach (headless WebGL rendering directly in Node.js using `gl@8` + `node-canvas-webgl` on Node 22 LTS). No browser process is launched, eliminating this issue entirely.

## 7. Prismarine-Viewer Web Server Exposed Without Auth (Puppeteer Approach)

**Severity: Low–Medium**
**Status: Resolved (approach changed)**

### Description

The previous puppeteer-based implementation started a prismarine-viewer web server on a hardcoded port.

### Resolution

The current node-canvas-webgl approach renders entirely in-process using headless WebGL. No web server is started for the viewer, eliminating this attack surface.

## 8. Screenshot Data Leaking to Disk via Event Recording

**Severity: Low**
**Status: Mitigated (but fragile)**

### Description

The `EventRecorder.record()` method filters out screenshot events before writing to disk:

```python
events_to_save = [e for e in events if e[0] != "screenshot"]
```

However, this filtering depends on the exact string `"screenshot"` as the event type key.

### Risk

- If the event key is changed on the Node.js side without updating the Python filter, base64 image data will be persisted to the `ckpt/events/` directory in JSON files.
- The `resume()` method reads these files back — if screenshots leak to disk, they would be loaded into memory on resume.
- The `init_position` extraction in `record()` (`events[0][1]["status"]["position"]`) assumes the first event has a dict payload. If event ordering changes and a screenshot event appears first, this will crash or produce incorrect results.

### Mitigation

- Use a constant or enum for event type names shared between Node.js and Python rather than magic strings.
- Add a defensive check: reject any event whose payload is a string longer than a threshold (e.g., 10KB) from being written to disk.

## 9. Unbounded Base64 Payload Size

**Severity: Low–Medium**
**Status: Current**

### Description

There is no limit on the size of the base64 screenshot string. The Express server has a 50MB body parser limit, and the screenshot is embedded inline in the JSON response. A 512×512 JPEG at quality 80 is typically 30–80KB base64, but there is no enforcement.

### Risk

- If resolution constants (`WIDTH`, `HEIGHT`) or JPEG quality are changed (or if the capture fails partially and produces an oversized buffer), the payload could grow to megabytes.
- Large payloads increase the LLM API request size, potentially hitting token limits, increasing latency, and raising costs.
- Memory pressure from multiple large in-flight base64 strings.

### Mitigation

- Add a size check in `screenshot.js` before returning the base64 string. If it exceeds a threshold (e.g., 200KB), either re-compress or return `null`.
- Add a corresponding check on the Python side when extracting the screenshot.

## 10. Dependency Supply Chain Risk

**Severity: Medium**
**Status: Prospective**

### Description

The vision pipeline introduces new dependencies with significant attack surface:

| Approach | New Dependencies | Concern |
|---|---|---|
| **Puppeteer** | `puppeteer` or `puppeteer-core`, Chromium binary (~200MB) | Full browser engine; large attack surface; auto-updates from Google CDN |
| **node-canvas-webgl** | `gl` (native), `node-canvas-webgl`, `three` (already present) | Native C++ code compiled from source; `gl` bundles ANGLE which has had CVEs |

### Risk

- `puppeteer` downloads a Chromium binary from `storage.googleapis.com` at install time. A compromised CDN or MITM during download could inject a malicious binary.
- `node-canvas-webgl` compiles native code (`gl` → ANGLE) — build-time supply chain attacks are possible via compromised tarballs.
- `prismarine-viewer` loads client-side JavaScript over HTTP (to the local viewer page) which puppeteer then executes.

### Mitigation

- Pin exact versions of `puppeteer`/`puppeteer-core` and `prismarine-viewer` in `package.json` (avoid `^` ranges).
- Use `npm audit` regularly and integrate it into CI.
- For puppeteer, verify the Chromium download hash (puppeteer supports `PUPPETEER_DOWNLOAD_BASE_URL` for self-hosted mirrors).
- For node-canvas-webgl, prefer prebuilt binaries over compiling from source when available.

## 11. Vision Fallback Logic Can Be Externally Triggered

**Severity: Low**
**Status: Current**

### Description

The `_invoke_with_vision_fallback` method in `voyager.py` catches LLM errors and checks for keywords like `"image"`, `"vision"`, `"multimodal"`, or `"content_type"` to decide whether to disable vision:

```python
if "image" in err_str or "vision" in err_str or "multimodal" in err_str or "content_type" in err_str:
    self._vision_supported = False
```

### Risk

- An adversary who can influence LLM error messages (e.g., via a malicious proxy between Voyager and Ollama) could force vision to be disabled permanently for the session by injecting one of these keywords into an error response.
- Once `_vision_supported` is set to `False`, it is never re-enabled, even if the underlying issue was transient.

### Mitigation

- Implement a retry counter rather than a permanent disable. After N consecutive vision failures, disable for the current rollout but re-enable on the next `learn()` iteration.
- Log the exact error that triggered the fallback for post-hoc analysis.

## 12. Redundancy Removal Reduces LLM Grounding When Screenshots Fail Silently

**Severity: Low**
**Status: Current**

### Description

When a screenshot is provided, the action and critic agents omit textual observations for Biome, Time, Nearby blocks, and Nearby entities. However, if the screenshot is a corrupted/blank image (e.g., the viewer hasn't finished loading, the world hasn't rendered, or the capture returned a valid but meaningless JPEG), the LLM receives neither visual nor textual information about the environment.

### Risk

- The bot operates blind — the LLM generates code without knowledge of its surroundings, leading to nonsensical or dangerous actions.
- There is no validation that the screenshot actually contains meaningful content before stripping the textual fallback.

### Mitigation

- Validate the screenshot before stripping text: check file size (a blank/black JPEG is typically < 2KB at 512×512), or compute a simple entropy/variance metric on the raw pixels.
- If the screenshot fails validation, fall back to the full text observation as if no screenshot was provided.
